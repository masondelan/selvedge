Selvedge

An MCP server. v0.3.4. By Mason Delan.

Selvedge is a git blame for AI agents — but for the why, not just which line which model touched. The reasoning is captured live, by the agent, as the change is made. Six months later you can still ask "when was users.stripe_customer_id added and why?" and get a real answer.

What it looks like

$ selvedge blame user_tier_v2

  user_tier_v2
  Changed     2025-10-14 09:31:02
  Agent       claude-code
  Commit      3e7a991
  Reasoning   User asked to add a grandfathering flag for legacy
              free-tier users during the pricing migration. Stores
              the original tier so we can backfill discounts without
              touching billing history.

That reasoning was captured by the agent at the moment the column was added — written into Selvedge from the same context that produced the change. Not inferred from the diff afterward by a second LLM. Not a hand-typed commit message generalizing across a dozen files.

The mechanics

Selvedge runs as an MCP server. Your AI agent — Claude Code, Cursor, Copilot, anything that speaks MCP — calls six tools as it works:

ToolDoes
log_change Record a change with reasoning. The only writer.
diff History for an entity, or a prefix — users matches every users.*.
blame Most recent change to an entity. What, when, who, why.
history Filtered history with since, project, changeset.
changeset Every event sharing a feature slug — pull a feature's full scope across many entities.
search Full-text across paths, diffs, reasoning, agents.

Storage is local SQLite. Your data lives in .selvedge/ next to your code. No servers, no accounts, no API keys.

Setup

$ pip install selvedge
$ selvedge setup

selvedge setup detects which AI tools you have, writes the MCP entry into each tool's config, drops the canonical agent-instructions block into your project's CLAUDE.md / .cursorrules / copilot-instructions.md, runs selvedge init, and installs the post-commit git hook for commit-hash backfill. Every modified file gets a .bak next to it before any change reaches disk. Re-running on an already-set-up project is a no-op.

For CI bootstrap and devcontainer postCreateCommand, run selvedge setup --non-interactive --yes.

Why captured-live matters

There's a small but growing category of "git blame for AI agents" projects. The thing that distinguishes Selvedge:

Tool Reasoning source Granularity
Selvedge Captured live by the agent in context Entity (DB column, function, env var, dep, route)
Inference-after-the-fact tools Generated by a second LLM from the diff at session end Line
Commit-time tools Hand-typed commit message Line / commit

Captured-live matters because the agent's own intent is in the prompt context that produced the change. Inference-after-the-fact tools have to reverse-engineer that intent from the diff afterward — which is exactly the problem we're trying to solve. Entity-level matters because the question after git blame is usually "what's the history of this column?", not "what's the history of lines 40–48 of users.py?"

An empty reasoning field is itself a useful signal: the agent didn't have one. The reasoning quality validator surfaces empty, too-short, and generic-placeholder values at log_change time so you can correct your system prompt before the field gets permanently filled with "user request" entries.

What it doesn't do

  • Selvedge does not call an LLM. It is the storage and query layer; the agent supplies the reasoning.
  • It does not generate reasoning from diffs. If the agent didn't pass any, the field is empty.
  • It does not require Postgres, an account, an API key, or a network connection.
  • It is not a hosted service. Your .selvedge/ never leaves your machine unless you commit it.
  • It does not scan your codebase. Events are recorded by the agent at the moment of change, not by a periodic crawler.
